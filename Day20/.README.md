# Day 20 - Creating a Python Launch File in ROS 2

## Overview

Day 20 introduces **Python launch files in ROS 2**, which are used to start multiple nodes together from a single file. After learning how to create publisher and subscriber nodes in the previous day, this lesson shows how to avoid running each node manually in separate terminals by using a launch file instead.

This is an important ROS 2 workflow improvement because launch files make robotics projects easier to run, organize, and scale. Even in small projects, they save time, and in larger systems they become almost essential.

---

## What is a Launch File?

A **launch file** is a file used to start one or more ROS 2 nodes with a single command.

Instead of running commands like:

```bash
ros2 run <package_name> <node_name>
```

again and again for each node, a launch file lets you define multiple nodes in one place and launch them together.

This is especially useful when:
- A project has multiple nodes
- Several terminals would otherwise be needed
- You want a cleaner and more repeatable workflow
- You want to manage a robotics system more efficiently

---

## Why Launch Files Matter

The main reason launch files are important is simplicity.

Without a launch file:
- You need to open separate terminals
- You need to run each node manually
- You may forget a command or make mistakes

With a launch file:
- You run one command
- Multiple nodes start together
- The workflow becomes cleaner
- Development becomes more professional and scalable

This is why launch files are widely used in ROS 2 projects.

## Creating a Launch Folder

The lesson begins by creating a **launch** folder inside the package.

For example, inside a package such as `pub_sub`, you create a folder named:

```bash
launch
```

This folder is used to store ROS 2 launch files.

A typical structure starts looking like this:

```bash
pub_sub/
├── launch/
├── pub_sub/
├── CMakeLists.txt
└── package.xml
```

This helps keep launch-related files organized in the standard ROS 2 package structure.

---

## Important `CMakeLists.txt` Update

One of the most valuable points in the video is that simply creating a new folder is not enough. If you create a new folder like `launch`, you also need to make sure the package knows about it through `CMakeLists.txt`.

The lesson explains that when a new folder is added, it should be included properly so ROS 2 can detect and use it. The same idea would apply to other folders too, such as:
- `params`
- `maps`
- other custom resource folders

This is a very useful beginner lesson because many issues happen when the folder exists physically but is not properly handled in the package configuration.

---

## Creating the Python Launch File

Inside the `launch` folder, the lesson creates a Python launch file. The example file name used in the video is:

```bash
pub_sub_launch.py
```

The exact name can be different, but the idea is to create a Python file with a `.py` extension inside the `launch` directory.

This file contains the configuration needed to start multiple ROS 2 nodes together.

---

## Launch File Code Structure

The lesson explains the main parts of a Python launch file in a simple way.

A typical launch file includes:

- Import statements
- A function that returns a launch description
- Node configurations
- Package names
- Executable names
- Optional custom node names
- Output settings

The key idea is that a launch file defines which nodes should run and how they should run.

This gives ROS 2 a structured way to start multiple executables from one command.

---

## Understanding `LaunchDescription`

One of the important ideas introduced in the lesson is the **launch description**.

This is the part of the Python launch file where the nodes to be launched are listed. In simple words, it acts like a collection of instructions telling ROS 2 what should start when the launch file is executed.

So instead of launching one node at a time manually, you put multiple node definitions into the launch description and let ROS 2 handle them together.

---

## Defining the Nodes

Inside the launch file, each node is described with configuration details.

These details include:
- The package name
- The executable name
- An optional custom name
- Output behavior

For the Day 20 example, the launch file includes:
- A publisher node
- A subscriber node

These are the same nodes created in the previous lesson, and now both are launched from one file.

This makes the launch file practical and easy to relate to what was already learned.

---

## Understanding the `output` Setting

The lesson also briefly explains the `output` setting in the launch file.

This refers to the logs or messages shown in the terminal when the nodes run. In a beginner-friendly sense, it controls where the node output appears so that you can observe publishing and subscribing behavior directly in the terminal.

That is helpful because when both nodes are launched together, their output can still be monitored in one place.

---

## Running the Launch File

After creating the launch file and updating the package configuration, the lesson runs the launch file using the ROS 2 launch command.

The command pattern is:

```bash
ros2 launch <package_name> <launch_file_name>
```

For example:

```bash
ros2 launch pub_sub pub_sub_launch.py
```

This starts the configured nodes together from the launch file instead of running them individually.

This is the main practical result of the lesson.

---

## What Happens When It Runs

When the launch file runs successfully, both the publisher and subscriber start together. The publisher begins publishing messages, and the subscriber begins receiving them at the same time.

This is useful because:
- You can see communication working immediately
- You do not need multiple manual `ros2 run` commands
- You save time
- The project becomes easier to manage

The lesson highlights that this is exactly why launch files are so useful in ROS 2.

---

## Why This Saves Time

The video clearly explains that running nodes individually in separate terminals can become inconvenient, especially as the number of nodes grows.

For two nodes, it may still feel manageable. But for four or more nodes, doing everything manually becomes inefficient. A launch file solves that problem by allowing several nodes to start together through one command.

This is one of the reasons launch files are a standard part of real robotics workflows.

## Scaling Beyond Two Nodes

Although the Day 20 example uses two nodes, the lesson also points out that the same approach can be used for more nodes.

A launch file can start:
- Two nodes

This is important because real ROS 2 systems often contain many components running together, such as:
- sensor nodes

So even though the example is simple, the idea scales to much larger robotics systems.
