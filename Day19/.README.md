# Day 19 - Creating Python Nodes in ROS 2

## Overview

Day 19 focuses on creating **Python nodes in ROS 2 from scratch**. After learning the basic ideas of publishers, subscribers, topics, and Turtlesim, this day shows how to actually write your own ROS 2 Python nodes inside a package and run them properly.

This lesson is important because it takes you from using ready-made ROS 2 tools to building your own executable logic. That is a major shift in the learning journey, because now you start writing ROS 2 programs yourself.

---

## What This Day Covers

The video is structured around a practical workflow for creating Python-based publisher and subscriber nodes inside a ROS 2 package. It covers package setup, Python file creation, node code structure, package configuration, building the workspace, and running the nodes.

The main flow of Day 19 is:

- Create a package for publisher and subscriber nodes
- Create Python node files
- Write publisher code
- Write subscriber code
- Update `CMakeLists.txt` and `package.xml`
- Build the workspace
- Source the workspace
- Run the nodes

This makes Day 19 one of the first complete end-to-end ROS 2 coding sessions in the series.

---

## Creating the Package

The lesson begins inside the ROS 2 workspace, specifically in the `src` directory, because packages in ROS 2 are created there.

A package is created using a command like:

```bash
ros2 pkg create pub_sub --build-type ament_cmake
```

The package name may vary depending on your choice, but in the lesson the package is specifically meant for publisher and subscriber node practice. The important idea is that the package is created inside the workspace’s `src` folder so it becomes part of the ROS 2 project structure.

---

## Creating the Python Module Folder

One very important point in the lesson is that after creating the package, you should create a folder inside the package that has the **same name as the package**.

So if your package is named:

```bash
pub_sub
```

then inside that package you create:

```bash
pub_sub/
```

again as a Python module directory.

This is important because it helps Python recognize the package structure properly and allows the node files to be placed in the expected location.

---

## Creating Python Node Files

Inside that Python package folder, the lesson creates the Python files for the nodes. These include:

- `publisher.py`
- `subscriber.py`

The lesson also highlights another very important file:

- `__init__.py`

This file is needed so Python treats the folder correctly as a package. Even though it can look small and easy to ignore, it is an important part of the setup.

The video also mentions giving executable permission to the Python files so they can run properly later.

---

## Writing the Python Nodes

The coding part of Day 19 uses **Python** and the ROS 2 Python client library.

The lesson explains that the Python node code requires:

- `rclpy`
- `Node` from `rclpy.node`
- A message type such as `String` from `std_msgs.msg`

This is important because every ROS 2 Python node needs the right imports before you can create publishers, subscribers, or other ROS functionality.

---

## Shebang Line

One detail strongly emphasized in the lesson is the **shebang line** at the top of the Python files.

```python
#!/usr/bin/env python3
```

This line tells the system which Python interpreter should be used to run the script. The video explains that if this line is missing, the node may not run correctly, so it is an important beginner detail to remember.

That may seem like a small technical setup point, but it is actually one of the most common issues beginners face when Python scripts do not execute as expected.

---

## Publisher Node Structure

The lesson explains the publisher node by walking through the main ideas in the code rather than treating it like magic.

The publisher node does the following:

- Creates a ROS 2 node
- Creates a publisher
- Selects a message type
- Publishes messages on a topic
- Uses a timer to publish repeatedly
- Logs the published messages

In the example, the publisher sends **String** messages on a topic named something like:

```bash
simple_topic
```

The node publishes messages repeatedly, and the lesson shows that each message can include a counter value so you can clearly see continuous publishing happening in the terminal.

This is a very useful teaching method because it makes the flow visible rather than hidden.

---

## Subscriber Node Structure

The subscriber node is built to listen to the same topic as the publisher.

Its main responsibilities are:

- Create a ROS 2 node
- Subscribe to the topic
- Receive incoming messages
- Process them through a callback
- Print or log the received data

The key idea here is that the subscriber must listen to the **same topic name** used by the publisher. If the topic names do not match, communication will not happen.

This directly reinforces the concept from Day 17, but now it is being implemented in real code.

---

## Topic and Message Type Matching

The lesson makes it clear that successful communication depends on matching both:

- **Topic name**
- **Message type**

If the publisher sends `String` data on a topic, then the subscriber must also subscribe to that same topic using the same message type.

This is one of the most important practical communication rules in ROS 2. When nodes fail to communicate, checking these two things is often the first debugging step.

---

## Understanding `rclpy.init()`, `spin()`, and Shutdown

The lesson also explains several key ROS 2 Python functions that are part of the node lifecycle.

### `rclpy.init()`

This initializes the ROS 2 Python client library. It prepares the node system to start working.

### `rclpy.spin(node)`

This keeps the node alive and allows it to continue doing its work, such as publishing messages or listening for callbacks.

### `destroy_node()` and `rclpy.shutdown()`

These are used to cleanly stop the node and shut down the ROS 2 client library when execution ends.

These lifecycle functions are very important because they appear in almost every ROS 2 Python node you write.

---

## Why Logging Matters

The publisher and subscriber both display outputs in the terminal so you can observe the communication happening.

For example:
- The publisher logs when a message is sent
- The subscriber logs when a message is received

This is useful because it gives visible confirmation that:
- The publisher is running
- The subscriber is running
- The topic is correct
- The communication is working

That kind of feedback is extremely helpful when you are learning ROS 2 for the first time.

---

## Updating `CMakeLists.txt` and `package.xml`

One of the most valuable parts of Day 19 is that it does not stop at writing Python code. It also explains that your package configuration files must be updated properly.

The lesson specifically mentions:

- `CMakeLists.txt`
- `package.xml`

These files need the correct entries so ROS 2 knows how to handle the Python nodes and package dependencies. If these files are not set correctly, the nodes may not build or run properly.

This is a crucial lesson because beginners often focus only on the `.py` files and forget that ROS 2 packages depend on proper configuration too.

---

## Building the Workspace

After updating the code and package files, the next step is building the workspace.

```bash
colcon build
```

This command processes the workspace and applies the changes you made. Without building the workspace, ROS 2 will not properly register the new package and executables.

This step reinforces a central ROS 2 workflow habit:
- write code
- update package files
- build workspace
- source workspace
- run nodes

---

## Sourcing the Workspace

After building, the workspace must be sourced.

```bash
source install/setup.bash
```

This step makes the newly built packages and executables available in the current terminal session. Without sourcing, ROS 2 may not recognize the package or node you just created.

This is another beginner-critical step because many errors happen simply because the workspace was built but not sourced.

---

## Running the Python Nodes

After the build and source steps, the nodes can be run with `ros2 run`.

The lesson demonstrates running the publisher node in one terminal and the subscriber node in another. Once both are running, the subscriber starts receiving the messages published on the topic.

This is one of the most rewarding moments in the lesson because it shows a complete working ROS 2 communication example built from scratch in Python.
