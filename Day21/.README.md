# Day 21 - Creating a Python Turtlesim Node in ROS 2

## Overview

Day 21 brings together two things you learned earlier: **Python ROS 2 nodes** and **Turtlesim**. The goal of this lesson is to create a custom Python node from scratch that controls the turtle automatically, so you can move beyond keyboard teleoperation and start making the simulator behave through your own code.

This is an important practical step because it shows how ROS 2 concepts become useful when applied to simulation. Instead of only launching nodes or manually publishing commands, you now create a node that sends motion commands to Turtlesim programmatically.

---

## What This Day Covers

The lesson focuses on building a custom ROS 2 Python node specifically for Turtlesim. The main workflow is:

- Create a new package for the Turtlesim node
- Create the Python module folder
- Add the Python node file
- Write publisher code for turtle motion
- Use the Turtlesim velocity topic
- Run the Turtlesim simulator
- Run the custom node
- Observe the turtle moving automatically

This makes Day 21 a very hands-on lesson and a strong continuation of the previous ROS 2 coding practice.

---

## Creating a New Package

The lesson begins by creating a new package inside the ROS 2 workspace `src` folder. In the video, a separate package is created specifically for Turtlesim practice rather than reusing the previous publisher-subscriber package.

The package is created using a command like:

```bash
ros2 pkg create xcore_turtlesim --build-type ament_cmake
```

The exact package name can vary, but the key point is that the package is made specifically to hold the custom Turtlesim node. This is a good practice because it keeps the project organized and focused.

---

## Creating the Python Package Folder

After creating the package, the next step is to create a folder inside it with the **same name as the package**.

So if the package name is:

```bash
xcore_turtlesim
```

then inside that package, another folder with the same name is created. This is the Python module folder where the node file is placed.

This structure is important because it helps Python treat the folder as a package and makes the ROS 2 Python node setup work correctly.

---

## Important Files Inside the Package

The lesson also creates the important Python package file:

```bash
__init__.py
```

This file helps Python recognize the folder as a proper module. It may look small, but it is a necessary part of the package setup.

Then the actual node file is created. In the video, the node file name is something like:

```bash
move_turtle.py
```

The file is also given executable permissions so that it can run correctly later.

---

## Why This Node is Different

In earlier lessons, the focus was on general publisher and subscriber examples. Day 21 applies the same ROS 2 node-writing concept to a specific simulator: **Turtlesim**.

That means this node is not just printing or receiving simple text messages. Instead, it publishes motion commands that directly affect the turtle in the simulator window.

This makes the learning much more visual and satisfying, because the effect of the code becomes easy to observe immediately.

---

## Required Imports

The lesson explains that the Python node needs the correct ROS 2 and message imports before it can work.

The main imports used are:

- `rclpy`
- `Node` from `rclpy.node`
- `Twist` from `geometry_msgs.msg`

The `Twist` message is especially important here because it is used to send velocity commands for moving the turtle.

This is one of the first times in the series where a more robotics-specific message type is used in a custom Python node.

---

## Publisher Node for Turtlesim

The node created in Day 21 is a **publisher node**. Its purpose is to publish velocity commands to the Turtlesim motion topic.

The lesson explains the structure in a practical way:

- Create a node class
- Create a publisher
- Select the correct topic
- Use a timer
- Publish velocity commands repeatedly

This repeated publishing is important because the turtle must keep receiving movement commands over time in order to keep moving.

---

## Finding the Correct Topic

A very useful beginner point from the lesson is that topic names should not be guessed blindly. The video explains that the correct topic name can be found by running Turtlesim and checking the available topics using:

```bash
ros2 topic list
```

From there, the correct command velocity topic can be identified. In the Turtlesim example, the motion topic used is:

```bash
/turtle1/cmd_vel
```

This is a valuable practical reminder because learning how to inspect ROS 2 topics is just as important as writing the node itself.

---

## Using a Timer

The lesson uses a timer inside the node so that the movement function is called repeatedly after a fixed time interval.

A timer is useful here because:
- It allows regular publishing
- It keeps the turtle moving continuously
- It avoids needing manual repeated commands

In the example, the timer is set to trigger every short interval, such as 0.5 seconds. This means the node keeps sending movement instructions in a regular cycle.

That is an important ROS 2 pattern because timers are commonly used for periodic behavior.

---

## Understanding the `Twist` Message

The motion command is created using the `Twist` message, which is commonly used in robotics for velocity control.

The lesson sets values like:
- **linear.x** for forward movement
- **angular.z** for turning

This gives a direct connection to earlier motion concepts:

- Linear velocity moves the robot forward
- Angular velocity rotates the robot

For example:
- If `linear.x` is positive and `angular.z` is zero, the turtle moves straight
- If both are non-zero, the turtle moves in a curved or circular path

This is one of the most useful robotics ideas reinforced by the lesson.

---

## First Motion: Straight Movement

The first version of the node publishes motion with a positive linear velocity and zero angular velocity.

That causes the turtle to move forward in a straight line. In the video, the turtle eventually reaches the wall because no stopping condition has been added yet.

This is actually helpful for learning because it makes it clear that the node is continuously publishing commands exactly as written. The simulator behaves according to the logic you provide, no more and no less.

---

## Logging Messages

The lesson also includes terminal log messages to make the node’s activity visible.

For example, logs are used to indicate:
- that the node has started
- that the turtle is moving

This is useful for debugging because it confirms that the node is running and publishing commands even before you fully rely on the simulator window for feedback.

Logging is a small detail, but it is a very good habit in ROS 2 development.

---

## Running the Turtlesim Node and Custom Node

After writing the code, the lesson runs both:
- the standard Turtlesim node
- the custom Python node

The Turtlesim simulator must be running first so the velocity topic exists and the custom node has something to control.

Once both are active, the turtle starts moving based on the commands from the custom node. This is the moment where everything comes together:
- package creation
- Python node creation
- publisher logic
- topic usage
- simulation response

That makes Day 21 one of the most rewarding beginner ROS 2 exercises in the series.

---

## Modifying the Code for Circular Motion

The lesson then goes one step further by changing the angular velocity from zero to a non-zero value.

That simple change causes the turtle to move in a circular path instead of a straight line. This is a very effective demonstration because it shows how small changes in code can directly change robot behavior.

It also reinforces an important robotics principle:
- movement behavior comes from the chosen linear and angular velocity values

This is exactly the kind of intuition that becomes valuable later when controlling real robots.
