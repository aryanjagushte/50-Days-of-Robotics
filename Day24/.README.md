# Day 24 - Creating a Web Interface for ROS 2 Turtlesim

## Overview

Day 24 introduces a **web interface** for controlling Turtlesim in ROS 2. Instead of using the keyboard, joystick, or only Python motion logic, this lesson shows how to build a simple browser-based UI that can control the turtle from a laptop or even a mobile phone, as long as both devices are connected to the same network.[page:12]

This is a very useful step because it connects ROS 2 with web technology. It shows that robot control interfaces do not always need to stay inside the terminal or a local Python script — they can also be exposed through a browser-based interface.[page:12]

---

## What This Day Covers

The lesson creates a small project with three main parts:

- A browser-based HTML user interface
- A ROS bridge connection between web and ROS 2
- Turtlesim control through that web UI[page:12]

The basic idea is simple:
- the web page sends commands
- those commands are passed into ROS 2
- Turtlesim receives the movement instructions
- the turtle moves accordingly[page:12]

This makes Day 24 one of the most practical integration examples in the series so far.[page:12]

---

## Main Goal of the Project

The goal is to control the Turtlesim turtle using a web page with buttons like:
- forward
- backward
- left
- right
- stop[page:12]

The video explains that the same interface can be opened on:
- a laptop browser
- a mobile phone browser

The only required condition is that the controlling device and the ROS 2 system must be connected to the **same Wi-Fi or hotspot network**.[page:12]

That makes the project feel much closer to a real robotics control panel.

---

## Why This Project Matters

This project is important because it introduces the idea of **remote UI-based robot control**. Even though the example uses Turtlesim, the overall architecture is highly relevant to robotics systems where operators need to send commands through a dashboard, tablet, or mobile device.[page:12]

This lesson helps you understand:
- ROS 2 and web integration
- network-based access
- browser-based control systems
- a practical use of ROS bridge[page:12]

That is a big step beyond basic terminal-only workflows.

---

## Using ROS Bridge

The key technology introduced in this lesson is the **ROS bridge server**. The transcript explains that ROS bridge is required because it creates the connection between ROS 2 nodes and the web interface.[page:12]

In simple terms:
- ROS 2 by itself does not directly talk to a simple web page in this setup
- the bridge server creates that connection
- the web UI can then communicate with ROS topics through the bridge[page:12]

This bridge is what makes the project possible.

---

## Installing ROS Bridge

The first setup step shown in the lesson is installing the ROS bridge server package.[page:12]

The transcript mentions installing the **ROS Humble rosbridge server**, which is needed before launching the bridge and connecting the web interface to ROS 2.[page:12]

This is one of the most important preparation steps, because without ROS bridge the browser UI cannot communicate with the ROS 2 environment.

---

## Finding the IP Address

A major practical step in the lesson is getting the **IP address** of the machine running the ROS 2 setup. The transcript explains that this IP address must be inserted into the web interface code so that the browser knows where to connect.[page:12]

This means:
- your system’s IP address will be different from the one shown in the video
- you must use your own local network IP
- the same IP must be used when opening the interface from another device on the same network[page:12]

This is a very important detail, because if the IP address is wrong, the web interface will not connect properly.

---

## The HTML Web UI

The lesson uses an **HTML file** as the user interface. According to the transcript, this HTML code includes:
- the IP address
- the topic details
- the message type
- button controls for turtle movement[page:12]

The video also mentions that the code is added to GitHub so that learners can use it in their own systems.[page:12]

This is useful because it shows that even a simple HTML page can become a robotics control interface when connected properly to ROS 2.

---

## Running the HTML Server

After preparing the HTML file, the next step is to navigate to the folder containing that file and start a local server for it.[page:12]

The transcript explains that the HTML code is served through a command run from the terminal so the browser can open it properly.[page:12]

This is an important detail because simply keeping the file on disk is not the same as running it in a way that allows the browser to interact with the ROS bridge setup.

---

## Launching the ROS Bridge WebSocket Server

Once the HTML side is prepared, the lesson starts the ROS bridge WebSocket server. The transcript shows this being launched and notes that the server starts on a port, and that a specific terminal output line confirms the connection setup is ready.[page:12]

This part is especially important because it forms the real bridge between:
- the browser UI
- the ROS 2 system
- the Turtlesim node[page:12]

Without this server, button presses on the web page would not reach the turtle.

---

## Running Turtlesim

The Turtlesim node must also be started, because it is the target system being controlled. The transcript explicitly says that running the Turtlesim node is necessary because the goal is to control it through the interface.[page:12]

So by this stage, the project typically has:
- the HTML UI server running
- the ROS bridge server running
- the Turtlesim node running[page:12]

This is another good example of a multi-part ROS 2 project architecture.

---

## Opening the Interface in a Browser

After the servers are running, the final step is to open the web link in a browser. The transcript explains that this link can be pasted into a browser on the same laptop or on a mobile phone, as long as the device is connected to the same Wi-Fi or hotspot.[page:12]

Once opened, the web UI displays movement controls for the turtle.[page:12]

This is the moment where the project becomes interactive and very practical.

---

## Controlling the Turtle

The transcript shows the turtle being controlled through button presses such as:
- forward
- left
- right
- stop
- backward[page:12]

The explanation describes repeatedly clicking these buttons to move the turtle around in Turtlesim.[page:12]

This makes the result very clear:
- the web interface sends the command
- ROS bridge passes it to ROS 2
- Turtlesim responds on screen[page:12]

That is a very satisfying demonstration of ROS 2 and web integration.

---

## Same-Network Requirement

One of the most repeated and important points in the lesson is the **same network requirement**. The transcript clearly states that if you are using a mobile hotspot or Wi-Fi, the devices involved must all be connected to that same network for the setup to work.[page:12]

This matters because the browser needs network access to the machine hosting the HTML server and ROS bridge server.[page:12]

For beginners, this is one of the most practical troubleshooting points in the whole project.

### Tips for Success

- Make sure all devices are connected to the same Wi-Fi or hotspot.
- Double-check the local IP address before testing the web UI.
- Start the HTML server, ROS bridge server, and Turtlesim in the correct order.
- Test the interface first on the same machine before trying a phone.
- Use simple button controls first, then expand later if needed.
- If the browser UI does not work, check the network and ROS bridge connection first.
